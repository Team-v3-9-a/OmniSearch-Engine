# 🐛 Отчет об уязвимостях и багах в Video Engine

В ходе анализа кода видео-движка на Go были обнаружены следующие критические и важные проблемы безопасности, надежности и производительности.

---

## 1. 🚨 Критическая уязвимость: Path Traversal & Произвольное удаление файлов (Remote DoS/Data Destruction)
* **Файл:** `video-engine/internal/api/server.go`
* **Место:** Метод `handleProcess` и `processVideo`

### Описание:
Сервер принимает JSON-параметр `video_id` от клиента и использует его без какой-либо валидации для формирования пути к рабочей директории:
```go
workDir := filepath.Join(baseDir, videoID)
os.MkdirAll(workDir, 0755)
defer os.RemoveAll(workDir)
```

Если злоумышленник отправит запрос с `video_id` равным `../../../../` или `/`, то:
1. `filepath.Join` разрешит этот путь относительно корня (например, `/app/shared_media/../../../../` превратится в `/`).
2. `os.MkdirAll` попытается создаться.
3. **Критично:** При завершении обработки выполнится `defer os.RemoveAll("/")` (или другой важной папки внутри контейнера), что полностью сотрет все файлы в доступной файловой системе контейнера/хоста.

### Рекомендация:
Необходимо санировать `video_id`. Он должен состоять только из алфавитно-цифровых символов, знаков дефиса или подчеркивания (например, UUID), либо нужно проверять, что очищенный путь действительно является поддиректорией `baseDir`.
```go
// Пример валидации:
if !isValidVideoID(req.VideoID) {
    http.Error(w, "Invalid video_id format", http.StatusBadRequest)
    return
}
```

## 2. ⚠️ Высокая критичность: Отсутствие таймаута в HTTP-клиенте коллбека (Утечка горутин)
* **Файл:** `video-engine/internal/api/server.go`
* **Место:** Метод `sendCallback`

### Описание:
Для отправки коллбека на бэкенд используется дефолтный `http.Client`:
```go
client := &http.Client{}
resp, err := client.Do(req)
```
По умолчанию в Go `http.Client` имеет бесконечный таймаут (`0`). Если бэкенд зависнет, не закроет соединение или сеть будет работать нестабильно, горутина, обрабатывающая видео, зависнет навсегда. Со временем это приведет к утечке памяти и горутин (Goroutine leak) и полному падению сервиса.

### Рекомендация:
Явно задавать разумный таймаут для клиента:
```go
client := &http.Client{
    Timeout: 10 * time.Second,
}
```

## 3. 🟡 Средняя критичность: Игнорирование отмены контекста (FFmpeg)
* **Файл:** `video-engine/internal/audio/extractor.go`
* **Место:** Метод `Extract`

### Описание:
Процесс извлечения аудио запускается через `exec.Command("ffmpeg", ...)` без передачи контекста. 
Даже если клиент отменит запрос, или сервер начнет Graceful Shutdown, процесс `ffmpeg` продолжит выполняться на сервере до самого конца, тратя CPU и диск.

### Рекомендация:
Использовать `exec.CommandContext(ctx, ...)` для автоматического убийства процесса `ffmpeg` при отмене родительского контекста.

## 4. 🟡 Средняя критичность: Игнорирование отмены контекста при нарезке кадров (OpenCV)
* **Файл:** `video-engine/internal/video/sampler.go`
* **Место:** Метод `SampleFrames`

### Описание:
Нарезка кадров с помощью OpenCV (`gocv`) выполняется в тяжелом цикле:
```go
for {
    if ok := video.Read(&img); !ok || img.Empty() {
        break
    }
    // ... сохранение кадра
}
```
Метод `SampleFrames` не принимает `context.Context`. Если сервер выключается или процесс отменен, тяжелый цикл декодирования видео будет крутиться до конца файла.

### Рекомендация:
Передавать `ctx context.Context` в `SampleFrames` и проверять `ctx.Err()` на каждой итерации цикла:
```go
for {
    select {
    case <-ctx.Done():
        return 0, ctx.Err()
    default:
    }
    if ok := video.Read(&img); !ok || img.Empty() {
        break
    }
    // ...
}
```

## 5. 🟡 Средняя критичность: Игнорирование ошибок загрузки кадров на S3
* **Файл:** `video-engine/internal/s3/client.go`
* **Место:** Метод `UploadMedia`

### Описание:
В случае ошибки при загрузке аудио-файла метод сразу возвращает ошибку:
```go
_, err := c.minioClient.FPutObject(...)
if err != nil {
    return "", fmt.Errorf("ошибка загрузки аудио: %v", err)
}
```
Но при ошибке загрузки отдельных кадров сервер просто пишет лог и продолжает выполнение:
```go
_, err := c.minioClient.FPutObject(...)
if err != nil {
    log.Printf("[S3] Предупреждение: ошибка загрузки кадра ...: %v", err)
}
```
Если у нас сломается загрузка кадров (например, закончится место на S3 или отвалится сеть), видео перейдет в статус `PROCESSING_ML` и ML-движок попробует обработать несуществующие файлы, что приведет к каскаду ошибок на бэкенде.

### Рекомендация:
Если кадры критически важны для пайплайна (а они важны), нужно возвращать ошибку и останавливать процесс, либо собирать ошибки и ретраить загрузку.

## 🟢 Низкая критичность: Состояние гонки при одинаковых `video_id`
* **Файл:** `video-engine/internal/api/server.go`
* **Место:** Метод `processVideo`

### Описание:
Если клиент отправит два запроса с одинаковым `video_id` одновременно (например, при двойном клике на UI), они создадут одну и ту же временную папку `/app/shared_media/{video_id}`.
Один процесс начнет перезаписывать файлы другого, а `defer os.RemoveAll(workDir)` первого завершившегося процесса удалит файлы прямо во время работы второго.

### Рекомендация:
Генерировать уникальную временную директорию (например, `/app/shared_media/{video_id}_{random_run_id}`) для изоляции запусков, либо возвращать ошибку `409 Conflict`, если видео уже обрабатывается (для этого нужен механизм распределенных блокировок или статус в БД бэкенда).
